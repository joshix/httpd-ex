Migration report

A long story of short content

Published Sun, Sep 3, 2017 by Josh Wood

History

Many of the words and pictures here have been around long enough to stumble mumbling through generations of domain names and server styles, from hand-crafted hardware hooked to home DSL, to containers shipped to the Cloud. This site is like your grandmother’s sofa. It gets moved more often than it gets updated.

This is the boring part of that story, outlining all the different ways much the same content has been constructed and delivered.

Long ago, there was rain-king.com

  • The first inkling of this site was a handful of bespoke HTML pages stuck in a corner of Utopian.net’s earliest mail server1, briefly addressed as utopian.net/josh, then rain-king.com
  • A few moments of Movable Type (I was the test user for every stray Perl script)
  • WordPress site on Utopian.net shared hosting, 2004-2008
  • Oldest preserved post: “Inked,” 2005-11-13

  • Jxggl project, June 2013: Move content to Google Sites and Blogger after Utopian.net was acquired

Jxnu (JoshiX.com New Underneath)

  • Jxnu project, early 2016: Content converted and site built from Markdown with Hugo, served by Caddy from a Container Linux host, using rkt to run a container consisting of Caddybox + the site HTML
  • Jxnu live on joshix.com, 2016-06-12
  • Container transitions through ACI, Docker, and OCI formats

Jxnu 2

  • 2017-09-03: Evolved away the web server and the Linux container host to simply write Markdown, make HTML with Hugo, and deploy to the Firebase hosting service. Many of the components of the first “nu” iteration have been eliminated from the second:

    • Caddy
    • Caddybox
    • Jxnu container
    • rkt
    • Container Linux

Out of the lab

It was more than convenient to use tech I helped build at CoreOS. A personal site may be more traditional than functional these days, but running this one on containers, on rkt, and eventually on Kubernetes provided the germs of several demos and talks.

However, producing HTML files and uploading them to a managed hosting service reduces cost and increases reliability. No more maintaining a durable cluster, nor even a machine, nor even so much as an HTTPd for this site. You could almost call it “serverless”.
